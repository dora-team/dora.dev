Observação: o gerenciamento de dados de teste é um dos um conjunto de recursos que aumentam o fornecimento de software e o desempenho organizacional. Esses recursos foram descobertos pelo programa de pesquisa DORA State of DevOps (em inglês), uma investigação independente e acadêmica rigorosa sobre as práticas e recursos que impulsionam alto desempenho. Para saber mais, leia nossos recursos de DevOps.

O teste automatizado é um componente essencial das práticas modernas de entrega de software. A capacidade de executar um conjunto abrangente de testes de unidade, integração e sistema é essencial para verificar se seu aplicativo ou serviço se comporta conforme o esperado e se pode ser implantado com segurança na produção. Para garantir que seus testes estejam validando cenários realistas, é essencial fornecer dados realistas aos testes.

Os dados de teste são importantes porque são exigidos por todos os tipos de testes em todo o conjunto de testes, incluindo os manuais e automatizados. Bons dados de teste permitem validar jornadas comuns ou de alto valor, testar casos extremos, reproduzir defeitos e simular erros.

No entanto, é difícil usar e gerenciar dados de teste com eficiência. O excesso de confiança nos dados definidos fora do escopo do teste pode tornar seus testes frágeis e aumentar os custos de manutenção. Depender de fontes de dados externas podem gerar atrasos e afetar o desempenho do teste. A cópia dos dados de produção apresenta riscos, já que eles podem conter informações confidenciais. Para enfrentar esses desafios, você precisa gerenciar seus dados de teste com cuidado e estratégia.

Como implementar o gerenciamento de dados de teste

A análise feita pelo DevOps Research and Assessment (DORA) mostra que as equipes de sucesso abordam o gerenciamento de dados de teste com os seguintes princípios básicos:

  • Os dados de teste adequados estão disponíveis para executar conjuntos de testes automatizados completos.
  • Os dados de teste para conjuntos de testes automatizados podem ser adquiridos sob demanda.
  • Os dados de teste não limitam ou restringem os testes automatizados que as equipes podem executar.

Para melhorar os processos de gerenciamento de dados de teste, esforce-se para atender cada uma dessas condições em todas as suas equipes de desenvolvimento. Essas práticas também podem contribuir positivamente para a automação de teste geral e os recursos de integração contínua.

Armadilhas comuns no gerenciamento de dados de teste

Gerenciar dados de teste pode ser difícil. As armadilhas a seguir são comuns no gerenciamento de dados de teste:

  • Depender de dados de forma excessiva em testes. Em particular, os testes de unidade não devem depender de dados ou estado externo ao teste.
  • Usar uma cópia completa do banco de dados de produção, em vez de identificar partes relevantes ou importantes.
  • Não mascarar ou gerar hash de dados confidenciais.
  • Confiar em dados desatualizados ou que não são mais relevantes.

Formas de melhorar o gerenciamento de dados de teste

As práticas a seguir podem ajudá-lo a usar os dados de teste com mais eficácia e eficiência:

  1. Favoreça testes de unidade. Os testes de unidade devem ser independentes entre si e com qualquer outra parte do sistema, exceto com o código que está sendo testado. Os testes de unidade não devem depender de dados externos. Conforme definido pela pirâmide de automação de teste, os testes de unidade devem compor a maioria dos seus testes. Testes de unidade bem escritos executados em uma base de código bem projetada são muito mais fáceis de serem triados e mais baratos de manter do que os testes de nível superior. Aumentar a cobertura dos testes de unidade pode ajudar a minimizar sua dependência de testes de nível superior que consomem dados externos.

  2. Minimize a dependência de dados de teste. Os dados de teste requerem manutenção cuidadosa e contínua. À medida que suas APIs e interfaces evoluem, é preciso atualizar ou recriar dados de teste relacionados. Esse processo representa um custo que pode afetar negativamente a velocidade da equipe. Portanto, é recomendável minimizar a quantidade de dados de teste necessários para executar testes automatizados.

  3. Isole seus dados de teste. Execute seus testes em ambientes bem definidos com entradas controladas e saídas esperadas que podem ser comparadas às saídas reais. Verifique se os dados consumidos por um teste específico estão explicitamente associados a esse teste e não são modificados por outros testes ou processos. Sempre que possível, os testes devem criar o estado necessário como parte da configuração, usando as APIs do aplicativo. O isolamento dos dados de teste também é um pré-requisito para que os testes sejam executados em paralelo.

  4. Minimize a confiança nos dados de teste armazenados nos bancos de dados. Manter os dados de teste armazenados nos bancos de dados pode ser particularmente desafiador pelos motivos a seguir:

    1. Mau isolamento de teste. Bancos de dados armazenam dados de forma durável. Quaisquer alterações nos dados persistirão nos testes, a menos que sejam redefinidas explicitamente. Entradas de teste menos confiáveis dificultam o isolamento do teste e podem impedir a paralelização.
    2. Impacto no desempenho. A velocidade de execução é um requisito essencial para testes automatizados. A interação com um banco de dados é geralmente mais lenta e mais complicada do que a interação com dados armazenados localmente. Favoreça bancos de dados na memória, quando apropriado.

  5. Torne os dados de teste prontamente disponíveis. A execução de testes em uma cópia de um banco de dados de produção completo apresenta riscos. Pode ser desafiador e lento manter os dados atualizados. Como resultado, os dados podem ficar desatualizados. Os dados de produção também podem conter informações confidenciais. Em vez disso, identifique seções relevantes de dados exigidas pelos testes. Exporte essas seções regularmente e facilite o acesso a elas para testes.

Como medir o gerenciamento de dados de teste

À medida que sua abordagem para gerenciamento de dados de teste evolui, é importante avaliar seu progresso em relação aos princípios básicos descritos anteriormente.

  1. Os dados de teste adequados estão disponíveis para executar conjuntos de testes automatizados completos. É possível medir isso rastreando quanto tempo os desenvolvedores e testadores gastam gerenciando e manipulando dados para uso em conjuntos de teste. Também é possível capturar isso por meio de medidas perceptivas (ou seja, pesquisas) para perguntar às equipes se elas têm dados adequados para o trabalho ou se sentem que isso é uma restrição para elas.
  2. Os dados de teste para conjuntos de testes automatizados podem ser adquiridos sob demanda. É possível medir isso como a porcentagem de conjuntos de dados principais disponíveis, com que frequência esses conjuntos de dados são acessados e com que frequência são atualizados.
  3. Os dados de teste não limitam ou restringem os testes automatizados que as equipes podem executar. É possível medir isso como o número de testes automatizados que podem ser executados sem a necessidade de adquirir dados de teste adicionais. Também é possível capturar isso com medidas perceptivas (ou seja, pesquisas) para perguntar às equipes se elas sentem que os dados do teste limitam as atividades de teste automatizados.
