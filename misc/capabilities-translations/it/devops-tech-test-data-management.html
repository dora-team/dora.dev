Nota: la gestione dei dati di test è un insieme di funzionalità che incrementano la distribuzione del software e le prestazioni dell'organizzazione. Queste funzionalità sono state scoperte dal programma di ricerca State of DevOps di DORA, un'indagine indipendente e rigorosamente accademica sulle pratiche e sulle capacità che promuovono prestazioni elevate. Per scoprire di più, leggi le nostre risorse DevOps.

I test automatici sono un componente chiave delle moderne pratiche di distribuzione del software. La capacità di eseguire un insieme completo di test di unità, integrazioni e sistemi è essenziale per verificare che l'app o il servizio si comporti come previsto e può essere implementato in sicurezza in produzione. Per garantire che i tuoi test convalidino scenari realistici, è fondamentale fornire test ai dati realistici.

I dati dei test sono importanti perché sono obbligatori per tutti i tipi di test durante l'intera gamma di test, compresi i test manuali e automatici. Dati di test validi ti consentono di convalidare i percorsi degli utenti comuni o di valore elevato, di testare gli scenari di limite, di riprodurre i difetti e di simulare gli errori.

Tuttavia, è difficile usare e gestire i dati dei test in modo efficace. L'utilizzo eccessivo di dati definiti al di fuori dell'ambito di test può rendere i test fragili e aumentare i costi di manutenzione. Le dipendenze sulle origini dati esterne possono introdurre ritardi e influire sulle prestazioni dei test. La copia dei dati di produzione introduce un rischio perché potrebbe contenere informazioni sensibili. Per affrontare queste sfide, devi gestire i dati dei test con attenzione e strategicamente.

Come implementare la gestione dei dati per i test

L'analisi eseguita da DevOps Research and Assessment (DORA) mostra che i team di successo affrontano la gestione dei dati di test con i seguenti principi fondamentali:

  • Sono disponibili dati di test adeguati per eseguire le suite di test automatizzate complete.
  • I dati dei test per le suite di test automatizzate possono essere acquisiti on demand.
  • I dati di test non limitano né limitano i test automatici che i team possono eseguire.

Per migliorare i processi di gestione dei dati di test, cerca di soddisfare ciascuna di queste condizioni in tutti i tuoi team di sviluppo. Queste pratiche possono anche contribuire positivamente alle tue funzionalità di automazione di test e integrazione continua complessive.

Insidie comuni nella gestione dei dati dei test

Gestire i dati dei test può essere difficile. Le seguenti inconvenienti sono comuni nella gestione dei dati di test:

  • Affidabilità dei dati nei test. In particolare, i test delle unità non devono dipendere da dati o stati esterni al test.
  • Utilizzare una copia completa del database di produzione, anziché identificare parti pertinenti o importanti.
  • Non mascherare o eseguire l'hashing dei dati sensibili.
  • Se utilizzi dati obsoleti o non più pertinenti.

Modi per migliorare la gestione dei dati dei test

Le seguenti prassi possono aiutarti a utilizzare i dati di test in modo più efficace ed efficiente:

  1. Test delle unità di misura. I test delle unità devono essere indipendenti l'uno dall'altro e da qualsiasi altra parte del sistema, ad eccezione del codice testato. I test delle unità non dovrebbero dipendere da dati esterni. Come definito dalla piramida di automazione di test, i test delle unità dovrebbero costituire la maggior parte dei test. I test delle unità ben scritti eseguiti su un codebase ben progettato sono molto più facili da valutare e più facili da gestire rispetto ai test di livello superiore. Aumentare la copertura dei test delle unità può aiutarti a ridurre al minimo la dipendenza da test di livello superiore che consumano dati esterni.

  2. Riduci al minimo l'utilizzo dei dati del test. I dati dei test richiedono una manutenzione accurata e continua. Man mano che le API e le interfacce si evolvono, devi aggiornare o ricreare i dati di test correlati. Questo processo rappresenta un costo che può influire negativamente sulla velocità del team. Pertanto, è buona norma ridurre al minimo la quantità di dati dei test necessaria per eseguire test automatici.

  3. Isolare i dati del test. Esegui i tuoi test in ambienti ben definiti con input controllati e output previsti che possono essere confrontati con gli output effettivi. Assicurati che i dati consumati da un determinato test siano associati in modo esplicito a tale test e non vengano modificati da altri test o processi. Dove possibile, i test dovrebbero creare lo stato necessario come parte della configurazione, utilizzando le API dell'applicazione. Isolare i dati dei test è anche un prerequisito per l'esecuzione parallela dei test.

  4. Riduci al minimo l'utilizzo di dati di test archiviati nei database. Mantenere i dati dei test archiviati nei database può essere particolarmente difficile per i seguenti motivi:

    1. Isolamento del test scadente. I database archiviano i dati in modo durevole; eventuali modifiche ai dati verranno mantenute tra i test, a meno che non vengano reimpostate esplicitamente. Input di test meno affidabili rendono più difficile l'isolamento dei test e possono impedire il caricamento in contemporanea.
    2. Impatto sul rendimento. La velocità di esecuzione è un requisito chiave per i test automatici. L'interazione con un database è generalmente più lenta e più complessa rispetto all'interazione con i dati archiviati localmente. Preferisci i database in memoria, ove opportuno.

  5. Fai in modo che i dati di test siano immediatamente disponibili. L'esecuzione di test su una copia di un database di produzione completo introduce un rischio. Aggiornare i dati può essere complicato e lento. Di conseguenza, i dati possono diventare obsoleti. I dati di produzione possono contenere anche informazioni sensibili. Identifica, invece, le sezioni di dati pertinenti richieste dai test. Esporta regolarmente queste sezioni e rendile disponibili per il test.

Come misurare la gestione dei dati per i test

Man mano che il tuo approccio al test dei dati si evolve, è importante misurare i tuoi progressi rispetto ai principi fondamentali sottolineati in precedenza.

  1. Sono disponibili dati di test adeguati per eseguire suite di test automatiche complete. Puoi misurare il tempo trascorso da sviluppatori e tester per gestire e gestire i dati utilizzabili nelle suite di test. Puoi anche acquisire questo provvedimento con misure percettive (ovvero sondaggi) per chiedere ai team di disporre di dati sufficienti per svolgere il loro lavoro o se ritengono che questo sia un vincolo per loro.
  2. I dati dei test per le suite di test automatizzate possono essere acquisiti on demand. Puoi misurare questo valore come la percentuale di set di dati chiave disponibili, la frequenza di accesso a tali set di dati e la frequenza di aggiornamento.
  3. I dati dei test non limitano né limitano i test automatici che i team possono eseguire. Puoi misurarlo come il numero di test automatici che possono essere eseguiti senza dover acquisire ulteriori dati di test. Puoi anche acquisire questo tipo di dati con misure percettive (ovvero sondaggi), per chiedere ai team se ritengono che i dati di test limitino le attività di test automatici.
