Hinweis: Proaktive Fehlerbenachrichtigung gehört zu einer Reihe von Funktionen, die zu einer höheren Software-Bereitstellung und Unternehmensleistung führen. Diese Faktoren wurden im Rahmen des „State of DevOps“-Forschungsprogramms des DORA-Teams ermittelt, einer unabhängigen, akademisch anspruchsvollen Untersuchung der Praktiken und Faktoren, die zu einer hohen Leistung führen. Weitere Informationen finden Sie in unseren Artikeln zu DevOps.

Bei einer proaktiven Fehlerbenachrichtigung werden Benachrichtigungen generiert, wenn sich die beobachteten Werte den bekannten Ausfallschwellenwerten nähern, anstatt zu warten, bis das System einen Ausfall meldet oder Sie gar von Kunden erfahren müssen, dass eine Anwendung oder ein Service ausgefallen ist. Mit diesem Ansatz können Sie Probleme erkennen und beheben, bevor sie ernst werden oder sich auf den Nutzer auswirken. Die Studie DevOps Research and Assessment (DORA) 2014 (PDF) hat gezeigt, dass proaktives Monitoring ein entscheidender Indikator für die Leistung der Softwarebereitstellung ist. Laut DORA-Studien können Teams, die mit proaktiver Benachrichtigung arbeiten, Probleme schnell diagnostizieren und lösen. Wenn Fehler nicht vom internen Monitoring, sondern hauptsächlich von einer Quelle außerhalb des Betriebsteams gemeldet werden, z. B. vom Network Operations Center (NOC) oder gar von Kunden, leidet die Leistung.

So implementieren Sie die proaktive Fehlerbenachrichtigung

Benachrichtigungsrichtlinien verwenden. Sie sollten Fehlerbenachrichtigungen mit bestimmten Benachrichtigungsrichtlinien generieren. Benachrichtigungsrichtlinien definieren die Bedingungen, unter denen eine Benachrichtigung generiert wird, sowie den Kanal für diese Benachrichtigung. Weitere Informationen zum Generieren von Benachrichtigungen finden Sie unter Monitoring und Beobachtbarkeit.

Schwellenwerte verwenden. Benachrichtigungsrichtlinien sollten Schwellenwerte für die von Ihnen beobachteten Messwerte verwenden, die auf tatsächliche Probleme hinweisen. Monitoring-Schwellenwerte lösen Benachrichtigungsrichtlinien aus, die Benachrichtigungen generieren, wenn Messwerte die Schwellenwerte überschreiten.

Schwellenwerte sorgfältig wählen. Wählen Sie die Schwellenwerte so, dass Benachrichtigungen nur dann generiert werden, wenn der Schwellenwert tatsächlich ein Problem vorhersagt. Wählen Sie also nicht willkürlich irgendeinen Wert. Im Allgemeinen sollten Sie ermitteln, ab welchen Werten es zu Beeinträchtigungen für den Nutzer kommt. Wählen Sie dann einen bestimmten Prozentsatz vor Überschreiten dieses Werts, bei dem eine Benachrichtigung ausgelöst wird.

Sie können beispielsweise eine Benachrichtigung auslösen, wenn die durchschnittliche Antwortzeit für Seiten 20 % unter einem Schwellenwert liegt, der bekannterweise zur Verärgerung der Nutzer und zu Support-Anrufen führt.

Vorfallanalysen führen. Wenn Sie nach Vorfällen Analysen aufstellen, ermitteln Sie, welche Indikatoren den Vorfall hätten vorhersagen können. Beobachten Sie diese Indikatoren in Zukunft.

Benachrichtigungsstrategie planen. Wenn eine Benachrichtigung keine oder jedes Mal dieselbe Aktion erfordert, sollten Sie die Antwort automatisieren. Sie sollten auch die Anzahl der Benachrichtigungen für Ereignisse berücksichtigen. Eine Flut von Benachrichtigungen während eines Ereignisses kann eher ablenken, anstatt nützlich zu sein. Eine Vielzahl von Alarmen kann die Empfänger desensibilisieren. Dies wird als Alarmmüdigkeit bezeichnet und kann zu längeren Reaktionszeiten oder zum Übersehen von Alarmen führen. Prüfen Sie Benachrichtigungen regelmäßig und löschen Sie solche, bei denen keine Handlung möglich ist.

Möglichkeiten zur Verbesserung der Fehlerbenachrichtigung

Konfigurieren Sie Benachrichtigungen so, dass wichtige Teams frühzeitig benachrichtigt werden, wenn Probleme in Ihren Systemen auftreten. Und zwar lange bevor sie zum Network Operations Center (NOC) oder zum Kunden vordringen. Mögliche Taktiken:

  • Benachrichtigungen in Logging- und Monitoringsystemen auf angemessene Werte konfigurieren
  • Benachrichtigungen so konfigurieren, dass Personen und Teams benachrichtigt werden, die das Problem beheben können
  • Proaktives Monitoring des Systemzustands basierend auf Schwellenwertwarnungen, bevor Systemfehler auftreten
  • Proaktives Monitoring des Systemzustands basierend auf Warnungen über Änderungsraten
  • Dafür sorgen, dass nur relevante Benachrichtigungen ausgegeben werden und das Team nicht zu viele Benachrichtigungen erhält. Überlegen Sie genau, welche Benachrichtigungen irrelevant sind. Deaktivieren Sie irrelevante Benachrichtigungen und aktivieren Sie relevante Monitoring-Benachrichtigungen wieder. Es ist nicht empfehlenswert, alle Benachrichtigungen zu deaktivieren.

Möglichkeiten zum Messen der Fehlerbenachrichtigung

Die Instrumentierung des proaktiven Monitoring ist unkompliziert. Dazu müssen diese Komponenten erfasst werden:

  1. Der Umfang, in dem Fehlerbenachrichtigungen von Logging- und Monitoringsystemen erfasst und verwendet werden
  2. Der Umfang, in dem der Systemzustand mithilfe von Schwellenwertwarnungen proaktiv beobachtet wird
  3. Der Umfang, in dem der Systemzustand mithilfe von Änderungswarnungen proaktiv beobachtet wird.

Damit verschiedene Aspekte Ihres Systems erfasst werden, sollten Sie Messwerte auf mindestens zwei verschiedene Arten beobachten. Sie können beispielsweise einen Messwertschwellenwert festlegen, der Benachrichtigungen auslöst, wenn ein Messwert über eine bestimmte Dauer unter oder über einem Wert liegt. In Verbindung damit könnten Sie eine Änderungsrate festlegen, die Benachrichtigungen auslöst, wenn die Änderungsrate eines Messwerts höher oder niedriger als erwartet ausfällt.
