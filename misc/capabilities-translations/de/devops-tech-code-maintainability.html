Hinweis: Verwaltbarkeit von Codes ist eine der Funktionen, die die Softwarebereitstellung und Leistung verbessern. Diese Faktoren wurden im Rahmen des "State of DevOps"-Forschungsprogramms des DORA-Teams ermittelt, einer unabhängigen, wissenschaftlich exakten Untersuchung der Praktiken und Faktoren, die zu einer hohen Leistung führen. Weitere Informationen finden Sie in unseren Artikeln zu DevOps.

Es erfordert ziemlich viel Code, um die von uns entwickelten Systeme auszuführen: Das Android-Betriebssystem umfasst 12 bis 15 Millionen Codezeilen, das monolithische Code-Repository von Google enthält mehr als 1 Milliarde Codezeilen und eine typische Smartphone-App hat 50.000 Codezeilen.

Der Bericht "State of DevOps 2019" aus der Studie DevOps Research and Assessment (DORA) zeigt, dass die Fähigkeit von Teams, ihren Code effektiv zu pflegen, eine von mehreren technischen Praktiken ist, die positiv zum Erfolg bei Continuous Delivery beitragen.

Wenn Ihr Team die Verwaltbarkeit von Code gut beherrscht, sind folgende Punkte gegeben:

  • Falls erforderlich kann das Team ganz einfach Beispiele in der Codebasis finden, den Code anderer Personen wiederverwenden und den Code von anderen Teams ändern.
  • Das Team kann problemlos neue Abhängigkeiten zum Projekt hinzufügen oder zu einer neuen Version einer Abhängigkeit migrieren.
  • Die Abhängigkeiten des Teams sind stabil und verursachen nur selten Fehler im Code.

Diese Ergebnisse zeigen, wie wichtig es für Entwickler ist, Code in der gesamten Codebasis der gesamten Organisation zu finden, wiederzuverwenden und zu ändern sowie Verfahren und Tools für das Abhängigkeitsmanagement zu implementieren.

Die Verwaltbarkeit von Code ist eine Funktion, die eine organisationsweite Koordination erfordert, da sie darauf beruht, den Code anderer Teams suchen, wiederverwenden und ändern zu können. Die effektive Verwaltung von Abhängigkeiten ist bei der Arbeit mit großen Codebasen und großen Organisationen oft eine enorme Herausforderung. Tools, mit denen Probleme mit Abhängigkeiten vermieden oder die Folgen von Codeänderungen aufgezeigt werden können, können die Designentscheidungen und die Codequalität für alle Entwickler verbessern, wodurch sie schneller arbeiten und eine stabilere und zuverlässigere Software erstellen können.

Verwaltbarkeit von Code implementieren

In Bezug auf die Implementierung lohnt es sich, das Quellcode-Management und das Abhängigkeitsmanagement getrennt zu behandeln. Es ist aber auch möglich, dass eine Lösung (wie das von Google verwendete monolithische Repository oder Monorepo-Muster) beide Probleme lösen kann.

Zuerst das Quellcodemanagement. Folgendes wird ermöglicht, wenn jeder Nutzer Elemente in der Codebasis ganz einfach finden, wiederverwenden und an andere weitergeben kann:

  • Schnellere Bereitstellung: Damit Software schnell bereitgestellt werden kann, müssen Teams den Code des jeweils anderen ansehen und Änderungen vorschlagen können. Wenn Sie dies so einfach wie möglich gestalten, können Sie erlangtes Wissen im gesamten Unternehmen weitergeben. Außerdem können so Teams Änderungen an anderen Teilen der Codebasis vornehmen, wenn ihre Arbeit davon abhängig ist.
  • Höhere Stabilität und Verfügbarkeit: Bei einem Vorfall ist es wichtig, Änderungen an einem beliebigen Teil der Codebasis schnell zu finden und vorschlagen zu können.
  • Höhere Codequalität: Eine Refaktorierung von Code zur Verbesserung seiner internen Qualität erfordert häufig Änderungen an mehreren Teilen der Codebasis. Wenn dies schwierig ist, wird es unwahrscheinlicher, dass jemand eine Refaktorierung durchführt. Außerdem erhöhen sich die Kosten dafür. Einige Organisationen, einschließlich Google, führen teamübergreifende Projekte zur Codeverwaltung durch. Dabei gehen Einzelpersonen die Codebasis durch, um Elemente auf Verwaltungsebene zu reparieren. Dafür ist es notwendig, auf Code im gesamten Unternehmen zugreifen und diesen ändern zu können.

Damit Beispiele gefunden und der Code anderer Personen wiederverwendet werden kann, muss auf den Quellcode des gesamten Unternehmens problemlos zugegriffen und dieser durchsucht werden können. Die einfachste Möglichkeit diese Anforderung zu implementieren besteht darin, eine einzige Plattformen zur Versionsverwaltung für den gesamten Code der Organisation zu verwenden. Auch dann, wenn dieser Code auf mehrere Repositories innerhalb der Plattform aufgeteilt ist. Je mehr Plattformen zur Versionsverwaltung verwendet werden, desto schwieriger kann Code gefunden werden.

Einige Organisationen möchten Teile der Codebasis sperren, sodass nur Personen mit einem berechtigten Interesse die entsprechende Stelle in der Codebasis sehen können (Google ist ein Beispiel für eine solche Organisation). Im Idealfall sollte es sich um die Ausnahme anstelle der Regel handeln. Software sollte so entworfen werden, dass die Menge an vertraulichem Quellcode minimiert wird. In vielen Fällen reicht die logische Trennung von vertraulichem Code mithilfe des Zugriffssteuerungsmechanismus des Versionsverwaltungssystems aus.

Außerdem sollte es möglich sein, von anderen Teams verwalteten Code zu ändern. Solche Änderungen erfordern in der Regel die Genehmigung des Teams, das für die Verwaltung des betreffenden Codes verantwortlich ist. Mechanismen wie Pull-Anfragen, bei denen ein Zweig in der Versionskontrolle erstellt und dieser durch eine Genehmigung wieder zusammengeführt wird, können die Bereitschaft stärken, anderen Teams Änderungen vorzuschlagen. Gleichzeitig werden nicht autorisierte Änderungen verhindert und Maßnahmen zur Informationssicherheit wie z. B. eine Aufgabenteilung durchgesetzt.

Betrachten Sie als Nächstes Abhängigkeiten. Es muss Folgendes gegeben sein, damit Teams leichter Abhängigkeiten hinzufügen und aktualisieren sowie gewährleisten können, dass diese stabil sind und nur sehr selten Fehler im Code verursachen:

  • Bessere Sicherheit: Mit zunehmendem Alter der Abhängigkeiten ist es wahrscheinlicher, dass darin Schwachstellen entdeckt werden. Abhängigkeiten müssen immer auf dem neuesten Stand sein, insbesondere wenn Sicherheitslücken gefunden und gepatcht wurden.
  • Schnellere Bereitstellung: Wenn Sie von anderen Teams oder Organisationen entwickelte Bibliotheken verwenden, müssen Sie zur Ausführung dieses Jobs keinen eigenen Code schreiben. Wenn vorhandene Mechanismen gewährleisten, dass Abhängigkeiten stabil sind und selten Fehler im Code verursachen, können Sie mehr Zeit für das Programmieren und weniger Zeit für die Wartung aufwenden.

Das Abhängigkeitsmanagement ist ein häufiges Problem für Softwareentwicklungsteams. Abhängigkeiten über Anwendungen hinweg auf dem neuesten Stand und konsistent zu halten, ist komplex, zeitaufwändig und teuer. Viele Organisationen weisen dieser Aufgabe keine ausreichenden Ressourcen zu. Dieses Problem wird durch unzureichende Prozesse und Tools erschwert. Dies kann ein erhebliches Sicherheitsrisiko darstellen, wenn Sicherheitslücken in Abhängigkeiten entdeckt werden und die Anwendungen, die sie verwenden, aktualisiert werden müssen.

Es ist wichtig, Prozesse und Tools zu übernehmen und weiterzuentwickeln, die es Teams erleichtern, bekannte Versionen von Abhängigkeiten zu nutzen und diese schnell zu aktualisieren. Dies schließt automatisierte Continuous Integration (CI) und Tests ein, um festzustellen, ob neue Versionen von Abhängigkeiten wichtige Änderungen enthalten, und um die Versionen der verwendeten Abhängigkeiten schnell und einfach mit den Systemen zu korrelieren, die sie verwenden.

Es gibt zwei häufig verwendete Modelle, um Abhängigkeiten in die Software aufzunehmen: das Vendoring und die deklarativen Manifeste. Beim Vendoring wird entweder der Quellcode oder die Binärdatei jeder Abhängigkeit zusammen mit der Anwendung in die Versionskontrolle eingecheckt. Alternativ haben die meisten modernen Plattformen ein Tool zum Abhängigkeitsmanagement, das die Abhängigkeiten in deklarativen Manifestdateien verwaltet, die in die Versionsverwaltung aufgenommen wurden (z. B. pip von Python, npm von Node.js, CRAN von R and NuGet von .NET).

Unabhängig davon, ob Sie Vendoring oder Manifeste verwenden, sind die folgenden Aspekte besonders wichtig:

  • Rückverfolgbarkeit: Von jedem Paket oder jeder Bereitstellung muss die genaue Version jeder Abhängigkeit zurückverfolgt werden können, die zum Erstellen verwendet wurde. Ohne dies ist es unmöglich, Probleme zu beheben, die durch Änderungen an Abhängigkeiten verursacht werden.
  • Reproduzierbarkeit: Ihr Build-Prozess sollte so deterministisch wie möglich sein. Es ist äußerst mühsam Probleme beheben zu wollen, die durch einen Erstellungsprozess verursacht werden, der sich auf verschiedenen Computern unterschiedlich verhält.

Wie bei Google Verwaltbarkeit von Code implementiert wird

Google implementiert die Verwaltbarkeit von Code durch einen relativ ungewöhnlichen Ansatz. Bei diesem Ansatz müssen zwar Kompromisse eingegangen werden und er ist nicht für jeden geeignet. Er ermöglicht es den Teams jedoch effektiv, die in diesem Artikel beschriebenen Ziele zu erreichen.

95 % der Softwareentwickler von Google weltweit arbeiten an einer gemeinsam genutzten monolithischen Codebasis, die über ein zentrales Versionsverwaltungssystem mithilfe eines Entwicklungsmodells nach Baumschema verwaltet wird. Im Jahr 2016 umfasste die Google-Codebasis "ungefähr eine Milliarde Dateien und [hatte] einen Verlauf von ungefähr 35 Millionen Commits über das gesamte 18-jährige Bestehen von Google. Das Repository enthält 86 TB Daten, was etwa zwei Milliarden Codezeilen in neun Millionen eindeutigen Quelldateien umfasst."

Da der gesamte Code von Google in einem einzigen Repository gespeichert wird, kann der Code anderer Teams leicht gefunden und geändert werden. Google stellt eine interne Suchmaschine bereit, die die Suche in der gesamten Codebasis ermöglicht. Entwickler haben eine Vielzahl an Tools, mit denen sie Änderungslisten zur Prüfung und Genehmigung erstellen können. Tests werden für jede Änderungsliste ausgeführt und Änderungslisten können aktualisiert und kommentiert werden. Das Code-Review-Tool von Google kann, ähnlich wie viele andere Plattformen, Prüfer für jede Änderung vorschlagen. Jedes Verzeichnis im Google-Repository verfügt über eine OWNERS-Datei (Inhaber), in der die Personen oder Gruppen aufgelistet sind, die Änderungen an Dateien in diesem Verzeichnis genehmigen können (ähnliche Funktionen sind in GitHub, GitLab und Bitbucket verfügbar). Durch eine schnelle Suche, Genehmigungsvorschläge und automatisierte Tests wird das Vorschlagen von Änderungen, das Prüfen und die Zusammenarbeit sowohl unkompliziert als auch robust.

Mit diesen Tools sind umfangreiche Refaktorierungen, die mehrere Teile der Codebasis ändern, relativ einfach durchzuführen und können in kleinstmöglichen Schritten durchgeführt werden. Google hat Tools entwickelt, die das Vornehmen von Änderungen, die sich auf wichtige Abschnitte der Codebasis auswirken, weiter vereinfachen und automatisieren. Google Tooling enthält ein Build-System namens Blaze, mit dem Änderungen, einschließlich Abhängigkeiten, aus dem Quellcode erstellt und getestet werden. (Teile von Blaze wurden in der Form des Open-Source-Tools Bazel veröffentlicht.) Es ist für jeden bei Google einfach, an einem beliebigen Teil von Google Änderungen zu entdecken und solche vorzuschlagen. Das schließt die Infrastrukturkonfiguration mit ein, die sich ebenfalls im selben monolithischen Repository befindet. Es ist ganz einfach, den Code mit anderen zu teilen und wiederzuverwenden.

Google bietet auch Steuerelemente, mit denen Sie Abhängigkeiten von Open-Source-Software verwalten können. Erstens muss für jede bei Google verwendete Open Source-Software die Quelle im monolithischen Repository von Google eingecheckt werden. Zweitens kann immer nur eine Version einer bestimmten Bibliothek in die Versionsverwaltung eingecheckt werden. Drittens ist die gesamte Software statisch verlinkt und aus der Quelle erstellt. Abschließend werden jedes Mal, wenn sich der Quellcode einer Bibliothek ändert, Neuerstellungen und automatisierte Tests für jede Software ausgelöst, die diese Abhängigkeit nutzt.

Durch diese Steuerelemente, in Kombination mit der leistungsstarken CI-Infrastruktur von Google, können unsere Produktionssysteme mit neuen Versionen von Abhängigkeiten ganz leicht auf dem neuesten Stand gehalten werden. Sie tragen auch dazu bei, dass alle Systeme konsistente Versionen einer bestimmten Bibliothek verwenden. (Dadurch kommt es nicht zu einer "Diamantabhängigkeit". Bei dieser ist ein Produkt auf zwei Komponenten angewiesen, die wiederum auf unterschiedlichen Versionen einer gemeinsamen Bibliothek beruhen, wodurch das Produkt nicht erstellt werden kann.)

Der Kompromiss bei Googles Ansatz zur Verwaltung von Abhängigkeiten von externem Code besteht darin, dass es schwieriger wird, neue Abhängigkeiten hinzuzufügen (eines der wichtigsten Ergebnisse der Verwaltbarkeit von Code). Bei jeder neuen Abhängigkeit muss der Quellcode in das monolithische Repository von Google eingecheckt werden. Dies bedeutet, dass der Code sowohl anfänglich als auch im Rahmen von Upgrades geprüft und getestet werden muss. Diese Genauigkeit hilft jedoch dabei, zu verhindern, dass Code mit Sicherheitslücken in die Produkte von Google gelangt, und stellt sicher, dass in allen Abhängigkeiten Verantwortliche klar festgelegt sind.

Häufige Schwierigkeiten bei der Implementierung von Verwaltbarkeit von Code

Hauptsächlich wird durch die Toolunterstützung und die Organisationskultur verhindert, dass jeder den gesamten Code durchsuchen und ändern kann.

Die erste häufige Gefahr sind mehrere Repositories zur Versionsverwaltung oder Repositories zur Versionsverwaltung mit restriktiven Zugriffseinstellungen. Organisationen sollten eine einzige Plattform zur Versionsverwaltung haben, auf der der gesamte Code gespeichert ist. Der Standardzugriff sollte idealerweise jedem in der Organisation ermöglichen, eine Quelldatei anzuzeigen, mit der Möglichkeit, den Zugriff für vertrauliche Dateien einzuschränken. Es sollte auch eine Möglichkeit geben, die Versionsverwaltung zu durchsuchen.

Stattdessen beschränken Unternehmen in der Regel den Zugriff auf die Versionsverwaltung. Dies führt zu der zweiten Gefahr: Es fehlt an Tools und Prozessen, um Änderungen an Teilen der Codebasis vorzunehmen, auf die sie keinen Schreibzugriff haben. Dadurch kann es erheblich schwieriger werden Problemen zu beheben, die (oft versehentlich) von anderen Teams verursacht werden, von deren Code Sie abhängig sind. Es verhindert auch Refaktorierungen, die mehrere Teile der Codebasis betreffen.

Zur Lösung dieses Problems bieten einige moderne Versionsverwaltungstools die Möglichkeit, Änderungsanfragen für Teile der Codebasis zu senden, zu prüfen und zu genehmigen, auf die Nutzer keinen Schreibzugriff haben.

Selbst wenn Tools unterstützt werden, müssen Unternehmen mit der Bereitstellung und Durchsuchbarkeit von Codebasen innerhalb des Unternehmens und möglicherweise für Anbieter und Auftragnehmer vertraut sein. Dies ist möglicherweise nicht für Organisationen möglich, deren Repositories zur Versionsverwaltung erhebliche Mengen an vertraulichen Informationen und Code enthalten, die nicht zwischen Teams geteilt werden können.

Die Verwendung von binären Abhängigkeiten ist weitaus häufiger, aber jede Sprache verfügt über eine eigene Toolchain zum Verwalten von binären Abhängigkeiten. Es ist ein komplexes Unterfangen, eine Standardstrategie sowie Konventionen zu erstellen, mit denen Abhängigkeiten über diese Toolchains im gesamten Softwareportfolio des Unternehmens effektiv verwaltet und nachverfolgt werden können. Damit die regelmäßige Aktualisierung von Bibliotheken von Drittanbietern nachvollzogen werden können, sind erhebliche Investitionen in die CI-Infrastruktur erforderlich, mit denen neue Versionen von Bibliotheken problemlos auf Kompatibilität mit vorhandenen Systemen und auf Schwachstellen getestet werden können.

In der Praxis überlassen es die meisten Organisationen den Teams, ihre Abhängigkeiten zu verwalten. Mit sehr unterschiedlichen Ergebnissen. Infolgedessen ist es in der Regel äußerst mühsam, schnell und vorhersehbar zu reagieren, wenn in einer Bibliothek eine Sicherheitslücke entdeckt wird. Selbst die betroffenen Dienste zu finden kann in der Regel eine langwierige Aufgabe sein.

Verwaltbarkeit von Code messen

Hier sind einige einfache Ideen, um Ihnen den Einstieg in die Messung der Verwaltbarkeit von Code zu erleichtern:

  • Wie viel Prozent der Codebasis Ihres Unternehmens können durchsucht werden?
  • Was ist die durchschnittliche Vorbereitungszeit, um einen Teil der Codebasis zu ändern, auf den ich keinen Schreibzugriff habe?
  • Welcher Prozentsatz unserer Codebasis ist doppelt vorhanden? Welcher Prozentsatz wird nicht verwendet?
  • Wie viel Prozent der Anwendungen verwenden nicht die neueste stabile Version aller von ihnen verwendeten Bibliotheken?
  • Wie viele verschiedene Versionen gibt es in der Produktionsumgebung? Wie ist der Medianwert? Was ist ein gutes Ziel? Wie viele Versionen sind älter als ein Jahr?
  • Wie oft aktualisieren Teams ihre Bibliotheken? Wie lange dauert das?

Bei der Auswahl der zu berücksichtigenden Messwerte gibt es drei Anwendungsfälle:

  • Technische und Design-Altlasten verwalten
  • Änderungsmanagement (einschließlich Notfalländerungen)
  • Sicherheitslücken patchen

Angesichts der wachsenden Codebasis ist die technische Altlast ein wichtiges Anliegen. Es ist wichtig, Code umgestalten und neu gestalten zu können, wenn sich Organisationen und die Produkte, auf die sie und ihre Kunden angewiesen sind, weiterentwickeln. Bei großen Codebasen kann dies ohne nennenswerte Werkzeugunterstützung komplex und mühselig sein. Es ist auch wichtig, Code identifizieren zu können, der nicht verwendet wird, dupliziert wird, eine schlechte Testabdeckung aufweist oder Sicherheitslücken enthält. Als Erstes muss sichergestellt werden, dass Sie mit Ihren Tools Messwerte erstellen und verfolgen können, die Verbesserungsbereiche identifizieren. Außerdem müssen sie es erleichtern, sichere Maßnahmen ergreifen zu können.

Der zweite Schritt ist das Änderungsmanagement. Inwieweit hilft Ihnen Ihr Tool, die Auswirkungen dieser Änderung zu erkennen, wenn jemand eine Änderung an einem Teil der Codebasis vornimmt? Wenn ein anderes Team betroffen ist, wie schnell kann es Maßnahmen ergreifen, um das Problem zu beheben? Insbesondere, wenn die Lösung in einem anderen Bereich der Codebasis liegt. Wie lange dauert es, bis die erforderlichen Codeänderungen in die Codebasis übernommen, getestet und freigegeben werden, wenn eine Notfalländerung vorgenommen werden muss?

Richten Sie Messwerte ein und verfolgen Sie sie, damit Sie verfolgen können, wie lange Änderungen dauern, bis sie sich in Ihren Prozessen verbreiten. Identifizieren Sie dann Engpässe und arbeiten Sie an der Verbesserung Ihrer Prozesse. Fügen Sie gegebenenfalls eine Tool-Unterstützung hinzu. Achten Sie auf "Notfall"-Prozesse, die Validierungen und Genehmigungen umgehen, um schneller ausgeführt werden zu können. Ziel sollte es sein, dass Ihr regulärer Prozess sowohl zuverlässig als auch schnell genug ist, um im Notfall effektiv zu sein.

Das Patchen von Sicherheitslücken ist ein besonders wichtiges Szenario des Änderungsmanagements. Wie lange dauert es, Software zu finden und zu patchen, die die anfälligen Versionen der Bibliothek verwendet, wenn eine Sicherheitsanfälligkeit in einer Bibliothek entdeckt wird? Wenn Sie sich nicht sicher sind, sollten Sie regelmäßig Tests durchführen. Angesichts der enormen potenziellen Kosten für die Behandlung von Sicherheitsverletzungen sowie der Daten- und Codeexfiltration und der Häufigkeit solcher Angriffe lohnt es sich in der Regel, erhebliche Ressourcen aufzuwenden, um sicherzustellen, dass die von Ihnen verwendete Software von Drittanbietern auf dem neuesten Stand ist und problemlos aktualisiert werden kann, sollten Sicherheitslücken entdeckt werden.
