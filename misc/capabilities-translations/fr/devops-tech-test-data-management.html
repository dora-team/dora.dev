Remarque : La gestion des données de test est l'un des ensembles de fonctionnalités permettant d'améliorer la livraison de logiciels et les performances organisationnelles. Ces capacités ont été découvertes par le programme de recherche DevOps Research and Assessment (DORA), une investigation indépendante et rigoureuse concernant les pratiques et les capacités permettant d'optimiser les performances. Pour en savoir plus, consultez nos ressources DevOps.

Le test automatisé est un élément clé des pratiques modernes de livraison de logiciels. La capacité à exécuter un ensemble complet de tests unitaires, d'intégration et système est essentielle pour vérifier que votre application ou service fonctionne comme prévu et peut être déployé en toute sécurité. Pour vous assurer que vos tests valident des scénarios réalistes, il est essentiel de leur fournir des données réalistes.

Les données de test sont importantes, car elles sont requises par tous les types de tests de votre suite de tests, y compris les tests manuels et automatisés. Les données de test de bonne qualité vous permettent de valider les parcours utilisateur courants ou à valeur élevée, de tester des cas spéciaux, de reproduire des erreurs et de simuler des erreurs.

Cependant, il est difficile d'utiliser et de gérer efficacement les données de test. Une dépendance excessive aux données définies en dehors du champ d'application du test peut fragiliser vos tests et augmenter les coûts de maintenance. Les dépendances sur des sources de données externes peuvent entraîner des retards et affecter les performances des tests. La copie de données de production présente un risque, car elle peut contenir des informations sensibles. Pour relever ces défis, vous devez gérer vos données de test avec soin et de façon stratégique.

Mettre en œuvre la gestion des données de test

L'analyse réalisée par DevOps Research and Assessment (DORA) (Recherche et évaluation) montre que les équipes performantes adoptent la gestion des données de test en suivant les principes fondamentaux suivants :

  • Des données de test adéquates sont disponibles pour exécuter des ensembles de tests entièrement automatisés.
  • Les données de test des ensembles de tests automatisés peuvent être obtenues sur demande.
  • Les données de test ne limitent pas le nombre de tests automatisés que les équipes peuvent exécuter ni leurs capacités.

Pour améliorer les processus de gestion des données de test, veillez à ce que toutes vos équipes de développement remplissent chacune de ces conditions. Ces pratiques peuvent également contribuer positivement à vos fonctionnalités globales d'automatisation des tests et d'intégration continue.

Problèmes courants liés à la gestion des données de test

Il peut être difficile de gérer les données de test. Les problèmes suivants sont courants dans la gestion des données de test :

  • Dépendance excessive aux données lors des tests. Plus particulièrement, les tests unitaires ne doivent pas dépendre de données ou d'états extérieurs au test.
  • Utiliser une copie complète de la base de données de production, plutôt que d'identifier les parties pertinentes ou importantes.
  • Ne pas masquer ni hacher les données sensibles.
  • S'appuyer sur des données obsolètes ou qui ne sont plus pertinentes.

Améliorer la gestion des données de test

Les pratiques suivantes peuvent vous aider à utiliser les données de test plus efficacement :

  1. Privilégiez les tests unitaires. Les tests unitaires doivent être indépendants les uns des autres et de toute autre partie du système, à l'exception du code testé. Les tests unitaires ne doivent pas dépendre de données externes. Comme le définit la pyramide d'automatisation des tests, les tests unitaires doivent constituer la majorité de vos tests. Les tests unitaires bien rédigés qui s'exécutent sur une base de code bien conçue sont beaucoup plus faciles à trier et moins coûteux à gérer que les tests de niveau supérieur. En augmentant la couverture de vos tests unitaires, vous pouvez réduire votre dépendance à des tests de niveau supérieur qui consomment des données externes.

  2. Réduisez la dépendance aux données de test. Les données de test nécessitent une maintenance minutieuse et permanente. Au fur et à mesure de l'évolution de vos API et interfaces, vous devez mettre à jour ou recréer les données de test associées. Ce processus représente un coût qui peut avoir un impact négatif sur la vélocité de l'équipe. Par conséquent, il est recommandé de réduire la quantité de données de test nécessaire pour exécuter des tests automatisés.

  3. Isolez vos données de test. Exécutez vos tests dans des environnements bien définis avec des entrées contrôlées et des sorties attendues pouvant être comparées aux sorties réelles. Assurez-vous que les données consommées par un test particulier sont explicitement associées à ce test et ne sont pas modifiées par d'autres tests ou processus. Dans la mesure du possible, vos tests doivent créer eux-mêmes l'état nécessaire dans le cadre de la configuration, à l'aide des API de l'application. L'isolation de vos données de test est également une condition préalable à l'exécution des tests en parallèle.

  4. Réduisez la dépendance aux données de test stockées dans des bases de données. La maintenance des données de test stockées dans des bases de données peut s'avérer particulièrement difficile pour les raisons suivantes :

    1. Mauvaise isolation du test. Les bases de données stockent les données de manière durable. Les modifications apportées aux données sont conservées entre les tests, sauf si elles sont explicitement réinitialisées. Des entrées de test moins fiables rendent l'isolation de test plus difficile et peuvent empêcher la parallélisation.
    2. Impact sur la performance. La rapidité d'exécution est une condition essentielle pour les tests automatisés. L'interaction avec une base de données est généralement plus lente et plus fastidieuse que l'interaction avec des données stockées localement. Privilégiez les bases de données en mémoire, le cas échéant.

  5. Mettez les données de test à disposition. L'exécution de tests sur une copie d'une base de données de production complète présente un risque. L'actualisation des données peut s'avérer difficile et lente à obtenir. Par conséquent, les données peuvent devenir obsolètes. Les données de production peuvent également contenir des informations sensibles. À la place, identifiez les sections de données pertinentes requises par les tests. Exportez ces sections régulièrement et rendez-les facilement disponibles pour les tests.

Mesurer la gestion des données de test

Au fur et à mesure que votre approche de la gestion des données de test évolue, il est important de mesurer vos progrès par rapport aux principes de base décrits précédemment.

  1. Des données de test adéquates sont disponibles pour exécuter des ensembles de tests entièrement automatisés. Vous pouvez mesurer cela en suivant le temps que les développeurs et les testeurs consacrent à la gestion et à la manipulation des données à utiliser dans les ensembles de tests. Vous pouvez également capturer cela par des mesures perceptives (c'est-à-dire par des enquêtes) pour demander aux équipes si elles disposent de données adéquates pour leur travail ou si elles estiment que c'est une contrainte pour elles.
  2. Les données de test des ensembles de tests automatisés peuvent être obtenues sur demande. Vous pouvez mesurer ce pourcentage comme le pourcentage d'ensembles de données clés disponibles, la fréquence d'accès à ces ensembles de données et la fréquence à laquelle ils sont actualisés.
  3. Les données de test ne limitent pas le nombre de tests automatisés que les équipes peuvent exécuter ni leurs capacités. Vous pouvez mesurer ce nombre comme étant le nombre de tests automatisés pouvant être exécutés sans avoir à acquérir de données de test supplémentaires. Vous pouvez également capturer cela avec des mesures perceptives (c'est-à-dire des enquêtes) pour demander aux équipes si elles estiment que les données de test limitent leurs activités de test automatisé.
